## 2025-05-18 - [Command Injection via execSync]
**Vulnerability:** Found multiple instances where user-controlled inputs (`projectDir`, `filePath`) were interpolated directly into shell commands via `execSync` (e.g., `execSync(\`node "\${cliPath}" init --dir "\${projectDir}"\`)`).
**Learning:** This is a classic command injection vulnerability. If `projectDir` contains shell metacharacters like `;`, `&&`, or `||`, it allows arbitrary command execution.
**Prevention:** Always use `execFileSync` (or `execFile` for async) instead of `execSync` when executing commands with dynamic inputs. Pass arguments as an array to ensure they are passed directly to the executable without shell interpolation.
## 2025-05-18 - [Command Injection via execSync with execFileSync Platform Differences]
**Vulnerability:** User-controlled inputs (`projectDir`, `filePath`) were interpolated directly into shell commands via `execSync` (e.g., `execSync(\`specify init --here --force \${aiFlag}\`)`).
**Learning:** `execFileSync` requires `shell: true` or manually specifying `.cmd` extensions for resolving npm binary wrappers on Windows (e.g. `specify` vs `specify.cmd`). Translating bash shell pipes (`| wc -l` or `| grep -c "DRIFT:"`) requires emulating the behavior via string manipulation and RegExp directly in JS (`.split('\n').filter(Boolean).length` or `.match(/DRIFT:/g)?.length`) instead of depending on shell utilities via `execFileSync`.
**Prevention:** Use `execFileSync` instead of `execSync`. When executing global npm packages via `execFileSync` across platforms, explicitly test for `process.platform === 'win32'` and append `.cmd` to the binary command name to prevent ENOENT errors on Windows, without falling back to shell interpolation.

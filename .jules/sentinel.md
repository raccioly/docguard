## 2025-05-18 - [Command Injection via execSync]
**Vulnerability:** Found multiple instances where user-controlled inputs (`projectDir`, `filePath`) were interpolated directly into shell commands via `execSync` (e.g., `execSync(\`node "\${cliPath}" init --dir "\${projectDir}"\`)`).
**Learning:** This is a classic command injection vulnerability. If `projectDir` contains shell metacharacters like `;`, `&&`, or `||`, it allows arbitrary command execution.
**Prevention:** Always use `execFileSync` (or `execFile` for async) instead of `execSync` when executing commands with dynamic inputs. Pass arguments as an array to ensure they are passed directly to the executable without shell interpolation.
## 2025-05-18 - [Command Injection via execSync - Part 2]
**Vulnerability:** Found remaining instances where `execSync` was used with dynamic input or unsanitized pipelines (e.g., `git log ... | wc -l`), bypassing standard shell execution boundaries.
**Learning:** `execSync`'s shell invocation creates critical risks even with non-obvious user data. When refactoring to `execFileSync`, one must natively recreate shell-like functions such as pipelines (`wc -l` counting became native `.split('\n').filter(Boolean).length`) and standard error suppressions (`2>/dev/null` became `stdio: ['pipe', 'pipe', 'ignore']`). Also observed that cross-platform CLI tool checks like `where` / `which` cannot use a shell pipe.
**Prevention:** Strictly enforce `execFileSync` universally except when deliberately and safely orchestrating known `.cmd` shims, and recreate all shell conveniences natively via JavaScript. Always ensure the `execFileSync` import is added when switching.

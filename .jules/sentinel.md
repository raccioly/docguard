## 2025-05-18 - [Command Injection via execSync]
**Vulnerability:** Found multiple instances where user-controlled inputs (`projectDir`, `filePath`) were interpolated directly into shell commands via `execSync` (e.g., `execSync(\`node "\${cliPath}" init --dir "\${projectDir}"\`)`).
**Learning:** This is a classic command injection vulnerability. If `projectDir` contains shell metacharacters like `;`, `&&`, or `||`, it allows arbitrary command execution.
**Prevention:** Always use `execFileSync` (or `execFile` for async) instead of `execSync` when executing commands with dynamic inputs. Pass arguments as an array to ensure they are passed directly to the executable without shell interpolation.

## 2026-05-26 - Command Injection via Unsanitized Configuration Output
**Vulnerability:** A command injection vulnerability existed in `cli/ensure-skills.mjs` and `cli/commands/init.mjs`. A dynamically parsed configuration value (`ai` from `.specify/init-options.json`) was directly interpolated into a shell string for `execSync`. An attacker could exploit this by supplying a payload like `"generic; touch /tmp/pwned;"`, resulting in arbitrary command execution.
**Learning:** Even when values are read from local files like configuration JSONs rather than direct user inputs, they can act as an attack vector if parsed dynamically and used in shell execution contexts. `execSync` executes through a shell and parses arguments using shell interpolation rules.
**Prevention:** Always validate configuration values against a strict allowlist regex (e.g., `/^[a-zA-Z0-9-]+$/`) before executing commands. Additionally, use `execFileSync` to pass arguments as an array instead of string concatenation, as `execFileSync` does not invoke a shell, bypassing interpolation attacks. Remember to add `.cmd` for Windows executables when using `execFileSync`.

## 2025-05-18 - [Command Injection via execSync]
**Vulnerability:** Found multiple instances where user-controlled inputs (`projectDir`, `filePath`) were interpolated directly into shell commands via `execSync` (e.g., `execSync(\`node "\${cliPath}" init --dir "\${projectDir}"\`)`).
**Learning:** This is a classic command injection vulnerability. If `projectDir` contains shell metacharacters like `;`, `&&`, or `||`, it allows arbitrary command execution.
**Prevention:** Always use `execFileSync` (or `execFile` for async) instead of `execSync` when executing commands with dynamic inputs. Pass arguments as an array to ensure they are passed directly to the executable without shell interpolation.

## 2025-05-18 - [Command Injection via execSync with stderr swallows and stdout piping]
**Vulnerability:** Found `execSync` used for seemingly safe commands like `which`, `where`, `git rev-parse`, and `git log`, but these commands often included shell-specific redirection like `2>/dev/null` or pipes like `| wc -l`. While some didn't use dynamic inputs, `execSync` inherently spins up a shell which can lead to unexpected behaviors or vulnerabilities if inputs are later added or altered. Passing user-controlled data into an executable string instead of an arguments array risks command injection.
**Learning:** `execSync` usage with shell features like pipes or redirection is inherently riskier than programmatic equivalents. Even for static commands, it sets a bad precedent and invites copy-pasting into vulnerable contexts.
**Prevention:** Always default to `execFileSync` (or `execFile` for async) over `execSync`, passing arguments as an array. To replace stderr swallowing (`2>/dev/null`), set the `stdio` option to `['pipe', 'pipe', 'ignore']`. To replace stdout piping (`| wc -l`), perform the line count natively in JavaScript (`output.trim().split('\n').filter(Boolean).length`).
